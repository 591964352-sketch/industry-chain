# 产业链数据 SOP · 通用模板（新增 / 刷新存量）· 四步版（v2，复查必经 + G8对齐检查必做）

> 适用于**任意产业链**。把 `<赛道id>`（pcb/hbm/semi…）、`<赛道名>`替换成目标链即可。PCB 是黄金范例，所有链对齐它。
> **v1→v2**：v1 三步版省去了人工复查层，导致光模块/商业航天/液冷等链反复出现"结构全、数据虚"——模板复制的是 PCB 的"长相"，复制不了把它填准的那轮联网核实。v2 把复查重新纳为必经环节，并与 SKILL.md 的「数据质量关卡 v2（G0-G7）」逐条挂钩。
> **v2→v2.1**：液冷链曾出现"G0-G7全部跑通+声明'整体收官'"，但实际S4b/S4c/S5/S8大量字段为空——**流程跑完 ≠ 数据齐全**。新增 **G8 PCB结构对齐检查**，作为CC声明"完成"或设 `meta.status: active` 前的强制关卡。

## 0. 流程（四步+G8，复查与对齐检查均必做）

```
CC 盘点(出清单+G0自检) → Gemini 查(三档输出+出注入块) → 复查(必经,按G7优先级)
→ CC 注入(守门+G7打标) → G8结构对齐检查(必做,0不达标才能设active)
→ 你拍板"通过" → commit → 复查纪要归档
```

页面右上角 🔄 刷新按钮生成的就是"盘点指令"。

**取舍说明（v1→v2）**：v1 试图靠"Gemini自查+CC守门"省去人工复查，结果微妙口径错/六维方向打反/伪填补Gemini自己查不出来、CC也拦不住（CC不联网，守门只能挡"无源/标存疑"，挡不住"看起来很像真数据但其实是拼凑/猜的"）。**复查是唯一不能省的一步**，省 token 的方式是优化复查的*效率*（见第 3 节"复查执行约定"），而不是省去这一步本身。

**G8说明（v2.1新增）**：G0-G7解决的是"数据是否核实过/方向是否打反"，G8解决的是另一个独立维度——"**数据密度/字段结构是否对齐PCB**"。两者都通过≠链真正完成。G8检查只对比结构不对比内容，CC本地遍历即可完成，几乎零token成本。

## 1. 角色分工

| 角色 | 联网 | 职责 |
|---|---|---|
| **CC** | ❌ | ① 盘点(出清单 + G0 自检标记) ② 注入(守门：G1/G2格式/G4三档格式/G5/G3结构性检查，不判断方向对错；并打 G7 优先级标签) ③ 结构/双校验。**绝不自己查数、绝不编造，G3方向对错不自行拍板"通过"。** |
| **Gemini** | ✅ | 核实每个数字(带源+tier+口径)，**自跑质检**，**三档输出**(①命中/②待补/③推算⚠️+逻辑)，直接输出 PCB schema 注入块。 |
| **复查端**（Sonnet/Opus/Haiku 分级） | 视情况 | **必经环节**：按 G7 优先级顺序核对，**G3 六维方向语义最终把关**（对照六维反向陷阱清单），核对③档推算逻辑，抓口径错/预测当实际。 |
| **你** | — | 传递 + 拍板"通过"；commit 后做"复查纪要归档"。 |

## 2. 不做假铁律

1. **CC 不查不编**：只出清单 + 注入已核数据，G3方向对错不自行判定。
2. **查必带源、必标档位**：每数有出处+三档之一（①命中/②待补/③推算⚠️）；关键数(市占/缺口/CAGR/PE)≥2 独立源、优先 primary/broker、排除单一 media。
3. **核不到留旧标待更新**，**绝不用估算覆盖真实数据**；③档推算必须附推算逻辑，不能只给结论。
4. **该刷才刷日期**：硬数据变才动"数据截止"日；纯六维主观(🆪) 不动。
5. **估值/PE 带 valAsOf**；注释 PE 带"(截至X日)"。
6. **复查不可省**：commit 前必须经复查端按 G7 顺序核对完毕。

## 3. 信源四档（TIER_RULES）+ 复查执行约定

- primary 🟢 一手：财报/公告/巨潮cninfo/交易所
- broker 🔵 券商·机构：券商研报/Prismark/协会
- media ⚪ 自媒体：股吧/财富号/雪球 → **单一来源关键数标"存疑(待核)"**
- estimate 🆪 推断：AI 主观/测算（六维分本身属此档）
- **⚠️推算非披露**：信源中无直接披露但可拼接推算得到的数字，**标记优先级高于上述四档**——即使底层引用是 primary/broker，只要结论本身是拼出来的，必须加 ⚠️ 并附推算逻辑（见 G4）

**复查执行约定（省 token）**：
- 复查端只读 Gemini 提炼好的紧凑块，不自己重新搜索
- 整条链一次过，按 **G7 优先级顺序**处理：`卡口` → `核心标的关键数` → `六维方向声明` → `推算⚠️` → `背景信息`
- 1-4 类逐条核对，5 类批量扫读、看着不对才单独搜该条
- **复查模型分级**：日常用 Sonnet；`卡口`类和核心标的关键数的最终收尾用 Opus；Haiku 只做机械清点（统计 G7 各标签数量、检查三档格式是否齐全，不做语义判断）

### A1.5 · CC 用 akshare 补 A 类数据（不联网，在 Gemini 查询之前执行）

> **定位**：A 类数据(财报 5 项)走 G4 ④档 akshare 直拉,**不经过联网端**。此步在 A1 出完待核清单之后、A2 发给 Gemini 之前执行,把 A 类数据**先于** Gemini 查完,缩小 Gemini 负担到纯 B 类。

**执行内容**：

1. 季报营收(¥ 精确到万元) + 同比(%)
2. 季报归母净利润(¥ 精确到万元) + 同比(%)
3. 季报毛利率(%)
4. PE-TTM 当前值(倍) + valAsOf
5. PE 历史分位(近 3 年自算) + asOf

**拉完后**：
- ✅ 成功拉到的字段 → 直接写入 `data/<id>.js`,tier: `primary`,src: `"akshare/新浪财经(基于公司季报)"`
- ❌ 失败/空数据 → 标 `"②待补(akshare 无数据)"`,不编造

**判定规则**：
- 报告期必须为当前最新一期(当前 = 26Q1),旧期数据不能冒充最新
- 每只票 `time.sleep(1)`,避免频率限制
- 不用训练知识填任何数字,查不到就 ②待补
- PE 亏损期为负 → 标 `"PE-TTM 失真(亏损期)"`,分位不填

完成 A1.5 后再进入 A2(发给 Gemini),Gemini 的查询范围已自动缩小为**纯 B 类数据**。

**首次确立**：`pcb` 2026-06-17 第三步,35/35 全字段完整走此流程(0 编造,5 只 PE 失真诚实标注,1 只 ②待补)。

### A2 · 给 Gemini（查 + 三档输出 + 出注入块 · 用第 6 节模板）
把 A1 清单(含【重点核查标记】) + 第 6 节模板发给 Gemini，它直接产出 PCB schema 注入块，**每条数据显式标三档之一**（①命中/②待补/③推算⚠️+逻辑）。`[高风险]` / `[高风险-方向]` 条目优先处理，即使查到①命中数据也保留该标记传给下一步。

### A3 · 复查（必经，按 G7 优先级顺序）
```
【全量刷新 <赛道id> · 第3步：复查 · 必经】
读取 Gemini 输出的注入块(紧凑块，不重新搜索)。按以下顺序核对：

1. 【卡口】chokePoints 相关条目 — 逐条核对数字、来源、verification 4项
2. 【核心标的关键数】segments 排名靠前个股的财务/产能关键指标 — 逐条核对
3. 【六维方向声明】dims6 各维方向，尤其命中 SKILL.md【六维反向陷阱清单】的环节 — 逐条核对：
   - supply: 供不应求→高分，过剩→低分
   - valuation: 5=最便宜(方向翻转，CC建链最容易写反，必查)
   - durability/visibility/policy/barrier: 高分=好
   - 每维 reason 是否有现实依据支撑(不能是空泛形容词)
4. 【推算⚠️】Gemini 标③档的条目 — 核对推算逻辑是否合理(不是核对数字本身，是核对"这样拼凑对不对")
5. 【背景信息】批量扫读，看着不对才单独搜该条

输出：标注每条"通过/需修正"+修正建议，交给CC注入。
```

### A4 · 给 CC（注入 · 不联网 · 守门 + G7打标）
```
【全量刷新 <赛道id> · 第4步：注入复查后数据 · 不联网 · 守门校验 + G7打标】
这是 Gemini 核实、复查端核对过的注入块。注入 data/<赛道id>.js 规则：

【守门 · 先过滤再写（结构性检查，不判断方向对错）】
  - 任何数据点缺"来源"/"tier"/三档标注 → 不写，保留旧值，列入"被拒清单"报我。
  - 标"②待补"或"存疑"的关键数 → 保留旧值 + 标"截至X日待更新"，不覆盖。
  - 单位/口径异常(如 PE 突然翻数倍、市占>100%、季度环比异常) → 不写，列入"存疑清单"报我。
  - dims6 各维 reason 缺现实依据/evidence → 打回，不写入(不进入下一步)。

【正常注入】
  - 通过守门的项：逐字段更新 值+asOf+tier。
  - 硬数据(财报/市占/缺口)有变 → 刷对应"数据截止"日；纯六维主观(🆪) → 不动日期。
  - 六维 dims6 若给了新值，按 PCB schema 更新；computeFit/个股分/擂台自动重算。

【关卡 v2 挂钩】
  - **G1 覆盖率自检**：注入后自动跑 `node` 离线脚本，分别输出：
    一手覆盖率"X/Y 个股带财报、缺 N 只：清单"(缺项票标 ⚠️财报待核)；
    推算占比"Z/Y 个股标⚠️推算非披露"(>30% 提示需专项核对)。
  - **G3 结构性检查**：确认每个方向声明都附带支撑依据/引用——**方向本身对错已在A3复查阶段确认，本步不重复判断**。
  - **G7 打标**：为每个条目打优先级标签(卡口/核心标的关键数/六维方向声明/推算⚠️/背景信息)，写入复查清单供后续追溯。

逐项报告 旧值→新值+tier+来源 + "被拒/存疑清单"，双校验(node自检/全链加载/G1覆盖率自检/抽查六维卡+擂台)，等"通过"再 commit。
```

## 6. Gemini 查询模板（通用 · 查 + 自查 + 三档输出 + 出注入块）

```
你是 A 股产业链投资研究的数据核查员。我要核「<赛道名>」。联网核实并严格遵守：

【纪律】
- 每个数字给来源(网站名+链接+日期)；关键数(财报/市占/缺口/PE)≥2 独立来源。
- 财报/估值类必须落一手(财报/公告/巨潮cninfo/交易所)，不能只用自媒体。
- 标 tier：primary/broker/media/estimate。仅 media 单一来源的关键数标"存疑(待核)"。
- 估值/PE 带"截至X日"。写清口径：国产自给率 vs 某厂市占？美元 vs 人民币？TTM vs 年度？

【三档输出（每条数据必须显式标档位，缺标视为格式不合规）】
- ①直接命中：数据 + 来源 + tier（一手/公开信源直接给出）
- ②待补：检索范围内未找到任何相关信息，不填数字，标"待补"
- ③⚠️推算非披露：信源没有直接给出该数据，但有相关信息可拼接/推算得到估算值——
  必须同时给出：推算结果 + "⚠️推算非披露"标记 + 推算逻辑(用了哪些数据、如何组合得出)。
  **不允许只给③档结论不给推算逻辑。**

【自查(输出前自己跑一遍，等于替我做质检)】
1. 有没有把财富号/股吧当成券商了？(纠正 tier)
2. 单位/口径有没有串？(¥/$、国产化率/市占、TTM失真)
3. 有没有遗漏重大风险？(实控人被查/减持/诉讼/订单不及预期)
4. 每个①档数字是否真有≥2 来源？(否则降为③档存疑或②待补)
5. 是否有条目本应是③档但被我当成①档写了？(重新检查推算痕迹)
6. 估值是否带 asOf？

【优先处理】清单中标 [高风险] / [高风险-方向] 的条目优先查证，即使查到①命中数据，
结果中也保留该标记，传给下一步复查端重点核对。

【要查的内容】
<粘贴 CC 的待核清单(含【重点核查标记】) / 或新增链则列环节+个股+赛道宏观>

【输出：直接给 PCB schema 注入块，每条标三档】
- 赛道级 prosperity.dims[6]：每维 {key,name,score(1-5),trend,reason(含硬数据+三档标注),
  evidence(来源+日期),flag,tier}；verdict 对象{longTermFit,oneLine,stockHint}。
  六维标尺：景气持续性/业绩可见度/政策确定性/供需紧张度/估值性价比(5=最便宜！方向翻转)/
  壁垒安全垫，各 1-5。
- 每只个股：{name,code,barrier档,dims6[6维 key/score/trend],dims6Note(含财报数+来源+三档标注),
  tier,valAsOf}；dims6 的 barrier 维分=barrier 档映射(极高5/高4/中高3/中2/低1)。
- 卡口：howToCheck + falsifySignal + valuation(pePercentile+asOf) + 风险点，三档标注。
- 每个数字旁标 tier + 三档标注 + 来源；存疑/③档项明确标出。不给买卖建议。
```

## 7. PCB 黄金范例 schema（对齐用）
- 六维：`prosperity.dims[]` 每维 `{key,name,score,trend,reason,evidence,flag,tier}`；`verdict` 为对象。⚠️**禁止压缩格式**(否则六维卡渲染失败)。
- 个股：每股 `dims6[]`+`dims6Note`+`tier`+`code`+`valAsOf`；dims6 的 barrier 维=barrier 档映射分。
- 综合分：`computeFitFromDims` 唯一权重(dur.25/vis.25/val.20/sup.12/bar.10/pol.08)；门控 valuation≤1 或 barrier≤1→封顶60；档 ≥75🟢/55-74🟡/<55🔴。
- 卡口：howToCheck+falsifySignal+valuation(pePercentile+asOf)；风险作 ⚠️ 提示、不改 barrier。
- 环节：材料→设备→制造→下游→侧枝；barrier 降序；制造商 choke:false 但有 dims6。
- 箭头：up绿/down红/flat灰；高分 flat 上色(score≥4→绿 / barrier 高档→金)。
