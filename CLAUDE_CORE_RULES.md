# CLAUDE_CORE_RULES.md · 产业链投研核心准则(横跨所有赛道)

> **本文件是横跨所有产业链赛道的核心方法论提炼。适用于任何新增产业链时套用,例如 [产业链名称 1]/[产业链名称 2]/[产业链名称 3]/...等等**。
>
> 与 CLAUDE.md 区别:
> - **CLAUDE.md**(本仓仓库专属)包含具体数据字段、本仓历史案例、所有 commit 归档
> - **CLAUDE_CORE_RULES.md**(本文件)只含跨产业链通用规则 → 可作为模板复制到新仓库
>
> 用法:
> 1. 新建一个产业链仓时,复制本文件作为 CLAUDE.md 基础
> 2. 在 §17(本文件中预留的扩展章节)填具体赛道信息
> 3. 加上 §6.7.1 防御章节的具体 prompt 范例
> 4. 逐渐沉淀出本赛道专属的 §11 待办清单格式

## §1 数据真实性铁律(0 级原则,适用于所有数据改动)

### 总原则

数据用于投资研究,**准确性高于一切**。**绝不为了流程跑完而违反纪律**(详细见 §8 数据准确度优先原则)。

### 三条不可逾越红线

| 红线 | 内容 | 违反后果 |
|---|---|---|
| **CC 不许造数** | AI 永远不得自行编造财报、市占、缺口、产能、价格 | 数据真实性失守 |
| **来源不得虚标** | 任何数字引用必须可核验(`L1/L2/L3/L4` 单源采信,`L5` 需两家以上) | §6.2 红线 + §8.1 红线 |
| **未核实的必须显式标注** | 联网搜不到 → 标 `estimate` 或保留旧值,绝不能填看似精确的数字 | 掩盖不确定性 |

## §2 信源分级体系(L1-L5 + estimate)

适用所有"会变的事实":市占率、缺口率、CAGR、产能、财报、PE 分位 等。

| 等级 | 来源类型 | 采信规则 |
|---|---|---|
| **L1**(primary) | 财报 / 公告 / 交易所 / 巨潮(cninfo) | 单源直接采信 |
| **L2**(监管 / 协会) | 政府部委 / 行业协会 / 监管数据 | 单源直接采信 |
| **L3**(专业机构) | Prismark / IHS / 弗若斯特沙利文 / SEMI / Yole / Gartner / TrendForce 等 | 单源采信,标注机构名+报告名+日期 |
| **L4**(头部券商) | 华泰 / 东吴 / 国金 / 东北 / 东方证券 等 | 单源采信,**只提供"券商名+报告标题+发布日期+关键数字",禁止提供任何 PDF URL** |
| **L5**(财经媒体) | 新浪 / 东方财富 / 雪球 / 财联社 / 今日头条 | **需两家以上媒体同一数字才采信** |
| **L6**(非公开) | 雪球用户帖 / 匿名调研纪要 / 未具名信源 | **直接拒绝,不得引用** |

### 渲染层兼容(已存档四档)

页面渲染字典 `SOURCE_TIERS` 只支持四档:**primary / broker / media / estimate**。
- `L1` → primary
- `L2`/`L3`/`L4` → broker
- `L5` → media
- AI 主观推断(如六维分、估值初判) → estimate
- 新增数据点时按上表选择 tier;缺省时返回空字符串,向后兼容

## §3 数据写入流程(7 步)

1. **明确 6 元组**:stock code + name + segment + rank + barrier tier + trend + trendNote
2. **禁止引入新 stock**:只有 prompt 列表里的 stock,禁止造其他 stock
3. **禁止伪造 URL**:包含 cninfo / dfcfw / 慧博 / iFinD / 任何 HTTPS 链接——LLM 无能力核验 URL,会编造
4. **7 段式输出**:
   - 【1. 认证进展】 / 【2. 客户切换】 / 【3. 新进入者】
   - 【4. 技术产能壁垒】 / 【5. 品类归属】 / 【6. 未查到】 / 【7. 信源清单】
5. **L1-L6 信源分层**(§2)+ 标注 `L<数字>:<来源类型>+<公告/报告标题>+<日期>+<关键数字>`
6. **数字带口径**:每个百分比必须显式标注(占比 / 同比 / 市占率 / 产能利用率 / 毛利率 / 良率 / PE 分位),不允许裸 `%`
7. **trend 独立判定**:trend 判断主依据必须是 A 类信号(认证 / 客户验证 / 市占 / 产能 / 技术),B 类(营收/订单/利润)只辅助印证

## §4 占比字段核实三件套

任何"某 stock 某业务/产品占公司总营收 X%"类数字必须先走 **akshare `stock_zygc_em`** 实测核验再写入。

| 步骤 | 内容 |
|---|---|
| 1 | 先用 akshare `stock_zygc_em` `MAINOP_TYPE=2`(按产品)+ `MAINOP_TYPE=1`(按行业)拉取主营构成 |
| 2 | 与豆包返回的占比数字对比,偏差 >5pp 视为幻觉,丢弃改用 akshare |
| 3 | 偏差 ≤5pp 标注 "⚠️ 口径差异" 保留豆包数字 |

akshare 数据通路 = emweb.securities.eastmoney.com,Windows 代理可通;**返回 GBK 编码,必须设 `r.encoding='gbk'`**。

akshare 不替代"产能 / 产能利用率 / 产量"——这部分仍须读年报 PDF + 强制窄范围复核。

## §5 强措辞使用规范

涉及**强承诺性表述**时必须慎重,因为投资场景下"锁单""独家""框架协议""定点"等措辞一旦失实,会误导实盘下单决策。

| 强措辞 | 强制要求 | 整改模板 |
|---|---|---|
| **锁单 / 长期供货协议** | 必须有 L1 年报/季报/专项公告披露 | 不写 → 改"客户验证推进 / 行业景气方向" |
| **独家供应** | 必须有 L1 + L3 双源,**且**全球厂家分布数据支持 | 不写 → 改"卡口环节之一" |
| **框架协议 / 定点协议** | 必须有 L1 法定公告披露 | 不写 → 改"客户批量供货 / 客户认证可视" |
| **唯一 / 唯二 / 寡头** | 全球厂家分布证据支持 + 认证周期 ≥18 月 | 不写 → 改"头部厂商之一" |
| **量产**(对应"已通过认证 + 批量供货") | 需 L1 公告披露批量金额或销量,非单一行业传闻 | 弱 → 仅"试生产"或"送样验证" |

**典型教训**:某次 bid 验证发现某 stock 的 position 字段被误加"国内市占 70%"等具体数字,经 akshare 实测 + cninfo 公告核对后该数字无据,整改时**改写为定性描述**(具体数字保留在 position/investableReason 字段以便未来追溯),凡未核实精度数字归【6. 未查到】或"具体数字待人工核实"。

## §6 日期核实清单(针对"具体日期"的强约束)

LLM 在日期类任务中**极易编造看似合理的具体日期**(例如把"2026Q3 小批量出货"细化为"2026-06-01")。

凡涉及**以下场景**必须强制走 4 类禁止:

| 场景 | 典型幻觉 |
|---|---|
| L1 公告认证日期 | 笼统"2026Q3 小批量" → LLM 细化为"2026-06-01 获官方认证"(无 L1 原文依据) |
| 设备销售单价 | "单台设备售价 600 万元(溢价一倍)"(实际数据字段完全无此表述) |
| 占比具体小数 | "AI 占主营业务 43.20%" vs "主营占比 93.74%" 嵌套混乱 |
| 早年历史数据 | "2021/2022 营收 XXX"(实际 <某年才上市> 无该年完整数据) |

**4 类禁止规则**:

1. **禁止**:具体认证日期(2026-MM-DD 形式)→ 归【6. 未查到】
2. **禁止**:具体设备销售单价 / 补贴金额 / 锁单金额 → 归【6. 未查到】或"具体数字待人工核实"
3. **禁止**:占比小数扩展(如"全球市占 26.9%" 是否包括 AI 服务器细分) → 限定嵌套口径
4. **禁止**:上市前的早年历史数据 → 限定"3 年完整可追溯"

**自查清单**(豆包 prompt 中显式要求):

| 检查栏 | 要求 |
|---|---|
| 高风险栏 | 主动声明"无删除原值,全程遵守 4 类禁止规则" |
| 中风险栏 | 主动软化"待 L1 原文核实" |
| 低风险栏 | 直接引用 + 标"position 字段·estimate" |
| 可重算栏 | 标"baostock L1 实测" + 给出具体算法 |
| 【6. 未查到】栏 | 完整覆盖 4 类禁止全部风险点 |
| trend 字段诚实判定 | 不沿用原 trend 值,独立判定 |

## §6.5 LLM 推理 vs 真实数据强约束(本节独立于 §6 日期核实)

> **本节为通用 LLM 幻觉防护,与 §6 日期核实并列**。任何需要"真实数据 / 接口调用 / 政策原文"的任务,必须强制要求 LLM(豆包/DeepSeek/Gemini)真实调用而非"逻辑推演 / 编造工具名"。

LLM 在以下场景**倾向用"逻辑推演"代替"真实数据拉取"**,看似输出"格式合规"但内容失真:

| 高风险场景 | 典型幻觉 | 真实情况 |
|---|---|---|
| **敏感性检验**(最近 N 天 PE 分位分布) | "12/18 分布"(看似合理) | baostock 实测"32/3"(完全相反) |
| **政策/政府数据**(目录入选 / 补贴金额) | "akshare.government_doc 接口返回 XXX"(虚构接口) | akshare/tushare/ifind 实际无政府政策类接口 |
| **具体日期细化**("2026Q3 小批量") | LLM 细化为"2026-06-01 获官方认证" | 无 L1 原文依据 |
| **具体金额编造** | "单台设备 600 万元"(凭空) | 数据文件完全无此表述 |
| **早年历史数据**(<某年才上市>) | "2021/2022 营收 XXX 亿" | 该年份无完整披露 |

### §6.5.1 强制防御规则(5 条硬约束)

1. **敏感性检验类任务强制实测验证**:
   - 任何"拉取最近 N 天数据 / 计算每个交易日 X / 统计多少天 >Y%" 类任务
   - LLM 返回后,**投顾必须用相同参数真实调用 baostock/akshare 重新执行一遍**
   - **不得仅凭 LLM 自称"已拉取"就采信** —— LLM 可能用"逻辑推演"伪装
   - 本规则为 §6.7.1 通用化版本

2. **关键警惕词 = 红色警告信号**:
   - LLM 报告中出现"逻辑推导""逻辑分析""推演""推算""估算"等关键词
   - **默认 LLM 未真实拉取数据**,必须触发投顾二次验证
   - 本规则为 §6.7.1 通用化版本

3. **格式合规 ≠ 内容真实**(核心):
   - LLM 输出格式合规(精确数字 / 合理分布 / L1-L6 信源分层)但**与已知数据有出入**
   - 即"看似合理但具体数字需要重新核验"的输出 → 必须投顾人工复核
   - 本规则为 §6.7.3 通用化版本

4. **政策类查询特殊处理**:
   - akshare / tushare / ifind / wind 等常用工具库中**目前均无政府政策类结构化接口**
   - 任何政策类查询的可靠数据源只有 L1(巨潮公告原文 / 工信部官网原文 / 财政部公告 / 官方 PDF)
   - LLM 返回的政策类数据若未指明**具体公告标题 + 发布日期 + 发布机构**,**默认视为虚构**
   - 本规则为 §6.7.2 通用化版本

5. **reason 字段"具体数字"3 类处理**:
   - **高风险(幻觉)**:彻底删除,改为定性描述或"待核实"
   - **中风险(待核)**:改为定性表述,标注"具体数字待人工核实"
   - **低风险(position 字段原有)**:直接引用 + 标"position 字段·estimate"
   - **可重算**:标"baostock L1 实测" + 给出具体算法
   - 本规则为 §6.7.3 通用化版本

### §6.5.2 prompt 设计硬约束

**任何涉及"拉取数据"的任务,prompt 必须显式要求 LLM 标注"数据来源 + 真实调用接口名 + 调用时间戳"**:

- 如不能给出 → 视为推演,标记为不可采信
- LLM prompt 中应**预先声明**该工具库的可用接口范围,例如:
  > "akshare / tushare / ifind / wind 工具库目前均无政府政策类结构化接口,请仅依赖 L1 政府公告原文 / 巨潮 / cninfo 实际公告回答"
- 若 LLM 仍返回"接口拉取" → 立即判 hallucination,整个 response 丢弃

### §6.5.3 反模式(禁止)

| 反模式 | 后果 |
|---|---|
| 仅凭 LLM "逻辑推导 / 推演 / 已拉取"措辞就采信 | 数字失真,推演被当作事实进入数据 |
| 不做投顾人工 baostock/akshare 二次验证就 commit | commit 标题 + 文件状态失真(违反 §10 + §13.X.2) |
| LLM 返回"akshare policy_doc 接口"就直接相信 | 接入虚构工具名,原因在于 akshare 根本没此接口 |
| 把"格式合规但内容失真"的输出默认通过 | 假数据进 canonical 文件,污染下游显示层 |
| 在 prompt 中未预先声明工具库无该接口 | LLM 默认推断一个看似合理的接口,导致幻觉 |

### §6.5.4 复用方式(11 链扩展)

任何 chain 套用本文件时:
- §6.5 通用,任何涉及"真实数据 / 接口调用 / 政策原文"的任务都适用
- 新 chain 使用的数据工具可能不同(例如 HBM 链可能用其他 L1 接口 / 行业 API),但**纪律完全相同**:强制实测 + 不信推演 + 投顾二次验证
- 新 chain 应在自己的 CLAUDE.md 中复制本节(§6.5.1 至 §6.5.3)

**违反本节 = §8 数据准确度原则 + §10 commit message 真实性双重违规**。

## §7 dims6 评分五档表(景气六维通用模板)

各赛道评分时引用本表;**特殊赛道(亏损 / 高景气 / 国产替代临界)允许有约束的弹性,必须显式标注**。

### 标准 5 档表

| 维度 | 5 分 | 4 分 | 3 分 | 2 分 | 1 分 |
|---|---|---|---|---|---|
| **景气持续性** | 3 年以上需求驱动 + L3 覆盖 + 下游锁单 | 1-2 年明确需求 + L3/L4 + 部分锁单 | 需求存在但周期性强,无锁单 | 需求疲软,产能过剩 | 需求收缩,库存高企 |
| **业绩可见度** | 年报/季报可见明确订单/框架协议 + L1 | L4 订单预测 + 客户公开验证 | L4 预测但无客户确认 | 仅有 L5 媒体 | 无可见订单 |
| **政策确定性** | 国家重点支持目录 + 专项补贴 + L2 | 行业政策 + L2 | 政策中性 | 政策收紧 | 政策明确不利 |
| **供需紧张度** | 全球缺口 >30% + L3 | 缺口 10-30% + L3/L4 | 供需基本平衡 | 供给略过剩 | 严重过剩 |
| **估值性价比** | PE 分位 <30% + 成长赛道历史低位 | PE 分位 30-50% | PE 分位 50-70% | PE 分位 70-85% | PE 分位 >85% 或历史极高位 |
| **壁垒安全垫** | 物理卡口(全球 ≤3 家)+ 认证 ≥18 月 | 认证壁垒 6-18 月 + 国内唯一/领先 | 壁垒存在但竞品 ≥5 家 | 壁垒低,竞争激烈 | 无壁垒,完全竞争 |

### 打分原则

1. **每个维度打分必须有数据依据**,不得凭感觉
2. **barrier ≤1 时综合分封顶 60**(质地硬约束不可绕过)
3. **barrier=极高但 q1=false 时**,在 fourQuestions 里加 note 字段说明原因(壁垒≠卡口)
4. **每次打分后 dims6.asOf 记录打分日期**(`YYYY-MM`)
5. **亏损公司专项规则**(§7.1)+ **估算 PE 失真 → 用 PB + PS + 业绩拐点**

### §7.1 亏损/经营异常公司专项规则(触发时强制生效)

触发条件:`L1 利润表最近 4 季度中 ≥1 季度净利润为负`或`最近 2 年度均为净亏损`或`PE-TTM 长期为负 >30 天`或`主营营收 CAGR <-10%`。

强制规则:

- **valuation 维度独立成条 query,禁止用 PE-TTM,必须用 PB(MRQ) 历史分位 + 亏损收窄趋势 + PS 横向对比**
- **query 前 baostock/akshare 预喂 L1 实证数据**(净利润正负 + 营收同比方向 + 实际 CAGR)
- **格式合规 ≠ 内容真实**:任何高风险维度的数字必须做 baostock/akshare 抽查验证
- **高风险数字特征识别**:精确小数点 / 整数百分比 / 全球整齐分布 / 单一信源覆盖 10+ 公司 → 必须要求豆包标注具体信源章节或列入【未查到】

## §8 数据准确度优先原则(0 级原则)

### 核心三原则(按优先级)

1. **数据准确度 > 流程完成** — 宁可任务不完成、数据保持上一稳定状态,也**绝不能为了"让流程跑完"而手动改字段或注入违规数据**
2. **严格遵守项目纪律** — 所有 CC 操作必须符合本文件 + §7 流程
3. **绝不为了流程跑完而违反纪律** — CC 推荐的所有方案必须先通过 §1 红线审查

### 反模式(禁止)

- ❌ "为了让 R-N 数据注入项目,CC 可以小改字段" — 违反 §1 + §8.1
- ❌ "手动修违规 URL 比重新跑模型快" — 违反 §1 + §8.3
- ❌ "流程上需要 commit,所以降低合规要求" — 违反 §8.1
- ❌ "用户可能不耐烦,所以走捷径" — 违反 §8.3
- ❌ "CC 代用户判断'哪些已修正'以节省 token" — 违反 §3.7 + §8.2
- ❌ "既然上一轮截断了,就下一轮只针对剩余违规" — 违反 §3.7(必须完整扫描)

### 验证清单(每次推荐方案前必走)

```
□ 1. 这个方案是否会让 CC 修改 LLM 返回的字段?
   - 是 → ❌ 拒绝(§1.2)
   - 否 → 继续 □ 2
□ 2. 这个方案是否会注入违反 §3.3 的字段(如 URL)?
   - 是 → ❌ 拒绝(§3.3)
   - 否 → 继续 □ 3
□ 3. 这个方案是否会让数据准确度降低?
   - 是 → ❌ 拒绝(§8.1)
   - 否 → 继续 □ 4
□ 4. 这个方案是否"为了流程跑完"而设计?
   - 是 → ❌ 拒绝(§8.3)
   - 否 → ✅ 可以推荐
```

## §9 barrier 维度跨环节引用禁令(<跨环节引用教训抽象代号>)

**barrier 维度禁止跨环节引用同业,必须严格在 stock **本细分赛道内**引用同业竞品**。

### 错误示范

某 stock 主营 `<主环节 idx>(<本细分赛道>)`,barrier 引用理由中**误引用**:
- 上游 `<上游环节 idx>(<上游细分>)` 厂商作为同业竞品
- 下游 `<下游环节>(<下游细分>)` 厂商作为同业竞品
- 隔环节 `<平行环节 idx>(<平行细分>)` 厂商作为同业竞品

### 正确做法

- **stock 主营 idx 内** 引用同业竞品(同环节 → 同环节内细分赛道 → 同细分赛道内同业厂商)
- 不要因为"上下游关联"或"同属一大行业"而模糊跨界
- 案例:某次 bid 验证发现某 stock 在 barrier reason 中误将"上游 X 厂商"列为同业竞品,整改后**严格限定在主营 idx 细分赛道内**,删除跨环节厂商名

### 强制 action

- 任何 reason 字段中提及"同业竞品 / 同业竞争者"必须**限定在该 stock 所在主营 idx 细分赛道内**
- 不可抗辩为"上下游同属 <产业链名称> 大行业,所以可互相引用"(这是红线)
- ack/diff 检查:`grep -E "同业竞品|同 segment" reason 文本` 必须命中主营 idx 内的厂商名,不允许命中跨环节 idx 内厂商名

## §10 commit message 真实性核查要求(<数据治理场景失真教训抽象代号>)

### 触发场景

任何**写入数据文件**的 commit,撰写 commit message 时若使用以下表述:

- "已完成" / "已补全" / "已有 reason 基础上维持" / "修复完成" / "全部通过"
- "N 处变更" / "M 项修改" / "X 个 stock 已校准"

### 强制要求

**commit 前必须先 grep 实际文件状态确认**,**不能凭印象撰写**:

1. **步骤 1**:用 awk/grep/python 提取实际文件状态(例如 stock 列表、dims6 reason 完整度、tier 字段值)
2. **步骤 2**:与 commit message 中声称的"已完成 / 已补全"对照
3. **步骤 3**:若发现不一致,**要么修正 commit message,要么重新跑 batch**
4. **步骤 4**:若已经失真而 commit 已 push,在下个 commit 中显式说明"上一 commit 标题描述与实际不符"并修正

### 禁止语句(直接 fail)

- ❌ "已有 reason 基础上维持"(实际文件无 reason)
- ❌ "8/8 reason 补全"(实际只有 7/8)
- ❌ "治理完成"(实际还有 3 个维度未审)
- ❌ "验证通过"(实际 baostock 拉取失败)

### 失真案例归档

| 失真类型 | 案例 | 后果 |
|---|---|---|
| 标题叙述与文件状态不符 | commit 标题写"已有 reason",实际 commit 时无 reason | 误导后续维护者,不知道该补哪里 |
| 完成数夸大 | commit 标题写"8/8 完成",实际只有 7/8 | 后续治理时跳漏 1 处 |
| 修复范围虚标 | commit 标题写"N 处",实际 git diff 只有 M 处(M < N) | 信源追溯失效 |

### 防御机制升级

- **commit 前置 gate**:用 `python scripts/page_audit.py` 第【N】项自动核对提交声称
- **dims6Audit 字段**:任何 `score=N` 的字段必须有 `reason` + `reviewedAt` + `reviewer`,缺一不可
- **每周回放 grep**:每周对前 10 个数据 commit 做"声称 vs 实际" diff,如有失真立即 fail 并修正

## §13.X 数据层同步纪律(commit 6.68 立 · 2026-07-07)

> **本节为 PCB 项目 33 个 commit(6.35-6.67)期间"修改 pcb.manual.js 但未同步检查 pcb.js / PCB_AUTO"导致的横幅误导性问题立**。所有 PCB 链及后续扩展的 11 链必须遵守本节同步纪律。

### §13.X.1 三层数据架构 + 同步关系

本仓数据架构分 **三层**,**任一层修改必须显式判断是否影响其它层**:

| 层 | 文件 | 生成方式 | 加载方式 | 影响范围 |
|---|---|---|---|---|
| **L1 · 手动层** | `data/<chain>.manual.js` | `scripts/edit_dim_field.js` 等工具手工编辑 | manifest 数组同步加载 | 六维 score/trend/tier/reason |
| **L2 · 合并层** | `data/<chain>.js` | 手工编辑(无自动生成脚本) | manifest 数组同步加载 | segments / midstream / overview / chokePoints |
| **L3 · 增强层** | `data/<chain>.auto.js` | `scripts/refresh_<chain>_valuation.py` baostock 拉取 | **可选**(manifest 数组可注释禁用) | 实时 PE-TTM / PE 分位 / 距前高回落幅度 + 横幅数据 |

### §13.X.2 commit message 强制说明项(硬约束)

**任何涉及数据字段的 commit,commit message 必须明确说明本次修改的数据层**:

```
✅ 标准写法示例:

[本次修改 pcb.manual.js L1 层]
- 修改文件:data/pcb.manual.js
- 影响数据层:L1 手动层
- 是否影响 L2(pcb.js)双层同步文件:否(本次仅修改 6 维字段,stock 列表未变)
- 是否影响 L3(PCB_AUTO)增强数据层:否(本次未涉及 PE-TTM / 估值字段)

[本次修改 pcb.js L2 层]
- 修改文件:data/pcb.js
- 影响数据层:L2 合并层
- 是否影响 L1(manual.js)双层同步文件:否(stock 列表变化,但 manual.js 已同步补全)
- 是否影响 L3(PCB_AUTO)增强数据层:否
```

**禁止默认省略**:
- ❌ 禁止 commit message 只描述"改了哪些字段",不说明影响哪些数据层
- ❌ 禁止默认假设"我只动了 manual.js 不影响其它" — 这是高频错误,必须显式写"不影响"才算说明
- ❌ 禁止对双层同步文件变化不验证 — §6.12 check_manual_pcb_sync 必须跑通才能 commit

### §13.X.3 修改 L1(manual.js)时的同步检查清单

修改 `data/<chain>.manual.js` 后,commit 前必须确认以下 3 项:

| 检查项 | 检查方法 | 是否需要执行 |
|---|---|---|
| 1. stock 列表是否变化? | `git diff data/pcb.manual.js \| grep -E "'[036]\d{5}':"` 查看是否新增/删除 stock | **变化 → 必须同步修改 L2 pcb.js** |
| 2. segments[idx] 是否变化? | `git diff data/pcb.manual.js \| grep -E "idx\s*:"` 查看 stock 段位变化 | **变化 → 必须同步修改 L2 pcb.js** |
| 3. 是否修改了 PE/估值相关字段? | `git diff data/pcb.manual.js \| grep -E "pe_ttm\|pePercentile\|fromHigh"` | **变化 → 必须评估 L3 PCB_AUTO 是否需要刷新** |

### §13.X.4 修改 L2(pcb.js)时的同步检查清单

修改 `data/<chain>.js` 后,commit 前必须确认:

| 检查项 | 检查方法 |
|---|---|
| 1. segments/midstream 股票列表与 L1 一致? | `python scripts/page_audit.py` 第【7】项必须 PASS |
| 2. 双层架构 stock 列表无悬空? | `node scripts/check_manual_pcb_sync.js` 输出"完全同步" |

### §13.X.5 修改 L3(PCB_AUTO)或 manifest 数组时的纪律

**L3 是可选层**(`pcb.auto.js` 在 manifest 数组中可被注释禁用),但**禁用/恢复必须有明确 commit**:

- **禁用** → commit message 必须写明:禁用原因 + 影响功能列表 + 是否计划恢复
- **恢复** → 必须先执行 `scripts/refresh_<chain>_valuation.py` 重新生成 L3 文件 + commit message 写明恢复日期
- **修改 manifest 数组** → 任何注释/取消注释都要 commit message 显式说明理由

**反模式(禁止)**:
- ❌ "临时禁用 [N] MB · 减重优化" 后 30+ 个 commit 没人想起恢复(典型 L3 manifest 注释禁用反模式 — 滥用临时操作累积成长期债务)
- ❌ 横幅显示硬编码历史 commit 标识,误导用户认为数据陈旧 — 真实状态是 manual.js 实时同步,只是 L3 文件未加载(信息展示必须基于当前 mtime,不能基于某个历史时刻)

### §13.X.6 数据层同步状态可视化(供 page_audit.py 第【8】项使用)

**本节定义 page_audit.py 第【8】项检测的数据层同步状态阈值**:

| 状态 | 判断标准 |
|---|---|
| ✅ 完全同步 | L1 mtime ≤ L3 mtime(增强层比手动层新或同期) |
| ⚠️ 提示性滞后 | L1 mtime 晚于 L3 mtime 但差距 ≤ 30 天 |
| 🚨 警告性滞后 | L1 mtime 晚于 L3 mtime 超过 30 天(但不阻断 commit) |
| ⛔ 阻断性失败 | L3 文件存在但 manifest 数组中注释禁用(本应启用却禁用 — 罕见) |

**🚨 警告性滞后** 是 **提示性**(不阻断 commit),仅在 `page_audit.py` 输出 WARNING,提醒人工评估是否需要执行 `refresh_<chain>_valuation.py`。

### §13.X.9 前端验证纪律(commit 6.69 立 · 2026-07-07)

> **本节为 §13.X 的延伸约束 — 专门针对 `index.html` 渲染逻辑(HTML/CSS/JS 展示效果)的"代码正确但未生效"陷阱**。

#### §13.X.9.1 核心原则(硬约束)

**任何涉及 `index.html` 渲染逻辑的修改(尤其是新增的 HTML/CSS/JS 展示效果),在 commit 前不得仅凭 sandbox 测试或代码逻辑推导判断为"已生效"。**

必须通过以下方式之一确认:

1. **(首选 · 真实浏览器验证)** 由用户在真实浏览器中硬刷 `Ctrl+Shift+R` 后,目视确认视觉效果符合预期,必要时附截图。
2. **(次选 · 代码执行路径证明)** 明确说明本次修改具体影响哪个渲染函数/哪个调用路径,并**代码层验证**该路径确实会被目标场景触发到。
   - **严禁**:"调用点存在" ≠ "调用点会被执行到"
   - 反例:**条件分支路径未执行**(如 `if (chainId === 'semi') ...` 中的代码,对 PCB 链永远不执行)
   - 反例:**循环条件过滤后无数据**(如 `segments.filter(s => s.name.includes('晶圆制造'))` 对无晶圆制造名称的 segment 永远是空)

#### §13.X.9.2 为什么需要这条纪律

`node` sandbox 测试 `conceptTagHtml({code:'300522'})` 返回 `'<span class="concept-tag" ...>⚠️ 概念票</span>'` 是**正确**的。但:

- 函数逻辑正确 ≠ 函数被调用
- 函数被调用 ≠ 调用点在目标渲染路径上
- 调用点在目标路径上 ≠ 目标场景会触发该路径

`page_audit.py` 的代码层检查也不能发现这类问题 — 它只能发现静态语法 / 硬编码颜色 / 函数名违规等,无法证明"运行时函数被特定调用路径真正调用"。

## §13.Y 数据真实性校验 + 检查脚本全覆盖纪律(本节独立于 §13.X 数据层同步纪律)

> **本节与 §6.5 配套**:§6.5 防 LLM 推理幻觉;本节防检查脚本虚假 PASS。两者是数据真实性两大支柱,缺一不可。

任何数据字段的"**真实性校验**"(数字 vs 真实世界数据)与"**检查脚本全覆盖**"(检查脚本扫过数据文件每个字段路径)是两件必须配对的事情。任一缺失都会让"假同步"+"虚假 PASS" 持续隐瞒真实数据错误。

### §13.Y.1 三类真实性校验缺失场景

| 场景 | 定义 | 典型反例(本规则抽象化) |
|---|---|---|
| **数字格式合法但内容错** | 6 位数字 / 百分比格式合法,但对应错误公司 / 错年份 / 错口径 | stock code 6 位数字合法但对应错误公司 / CAGR 数据格式正确但年份错位 |
| **检查脚本扫描范围不足** | 只扫"主路径",漏扫附加字段路径(如 midstream / treeMap / 上下游展示) | check 脚本只扫 `<chain>.js` 的 segments[*].stocks 漏扫其他 companies / treeMap 等路径 |
| **"完整 PASS"反成最危险信号** | 检查脚本报告"完全同步"但实际是漏检导致的虚假信号 | "page_audit 第【N】项 PASS"但实际是 10+ 处真实差异的隐藏 |

### §13.Y.2 强制规则(5 条硬约束,适用任何 chain)

1. **检查脚本必须先列"全部字段路径"再实现扫描**:
   - 任何索引/扫描逻辑的编写,**必须先在注释列出该数据文件所有可能存放关键数据的字段路径**
   - 逐个确认扫描覆盖(segments[*].stocks / segments[*].companies / midstream.stocks / treeMap.*[*].companies / 等)
   - **禁止**凭记忆 / 凭印象假设哪些路径有数据

2. **数据真实性校验分两层**(双硬约束):
   - **格式校验**(类型 / 长度 / 数值范围):必须 — 任何 chain 项目都应实装
   - **vs 真实数据校验**(数字 → 公司 / 年份 → 行业):**强烈建议** — 缺失时应登记为本仓 §11 待办

3. **page_audit 第【N】项输出必须是"差异清单 + 解决状态"两段**:
   - 不能只用"完全同步" / "0 项差异"做结论标志
   - 必须列出:**完整扫描了什么** + **找到哪些差异** + **每项的解决状态**(真实差异 / 白名单豁免 / 误报 / 已修复)
   - "PASS"必须在所有差异都已明确处理后才报

4. **commit 修复盲点必须配合"白名单豁免机制"**:
   - 盲点扩大扫描范围后,新发现的"差异"可能是**设计有意为之**(如渲染层附加展示 vs 核心池)
   - → 必须配套白名单豁免 + 完整登记理由(4 元组:code + name + 位置 + 豁免理由)
   - **禁止**"修复盲点后直接报 PASS" 模式 —— 必须先列差异清单 + 配套白名单

5. **三类真实性校验缺失必须显式标注**:
   - 任何 chain 项目首次跑检查脚本时,若发现"主路径 OK 但附加路径有差异":
   - **必须**标记为"未覆盖路径",不能视为 PASS
   - **必须**评估补充扫描路径 + 增加白名单

### §13.Y.3 反模式清单(任何 chain 禁止)

| 反模式 | 后果 |
|---|---|
| **检查脚本只扫"主路径"** | 真实数据差异被隐藏在未扫路径(如 midstream / treeMap / segments.companies 等) |
| **"完整 PASS"作为完成标志** | 维护者失去戒心,真实错误在生产环境持续 N 天存在 |
| **修复盲点后直接报 PASS** | 应扩大扫描 + 列出新增差异 + 白名单豁免有理由 —— 缺一不可 |
| **未列理由的"白名单"豁免** | 治理失去可追溯性,3-6 个月后无人记得为什么豁免 |
| **page_audit 仅报"完全同步"+"PASS"** | 用户无法判断哪些路径被覆盖、哪些差异被豁免 |
| **commit 标题写"已修复"+ 无 check 脚本支撑** | 违反 §10 commit message 真实性核查红线 + §8 数据准确度优先 |

### §13.Y.4 commit message 真实性核查联动(本节与 §10 协同)

任何 commit message 写"修改了 X 字段,全部 PASS" / "已修复" / "已校验" 措辞时,**必须真有过 check 脚本跑过**:

- 没有 check 脚本的字段不算"已修复"
- **三步骤不可跳过**:
  1. 先修检查脚本让它能检出该字段
  2. 再修实际数据(commit 该修复)
  3. 再跑 page_audit 确认无新增 FAIL + 三层数据架构完整一致

- **禁止**用"格式合规"代替"内容真实"(配合 §6.5)

### §13.Y.5 §13.Y.1-§13.Y.4 的复用方式

任何 chain 套用本节时:
1. **必须先盘点该 chain 的数据文件结构**:
   - 哪些字段路径可能存放 stock / code / 关键数字?
   - 有没有"渲染层附加展示" vs "核心池"的设计分层?
   - 对应设计白名单豁免规则
2. **必须修复** check 脚本使其扫描该 chain 的所有字段路径
3. **必须配套** 白名单豁免机制(若有"设计有意为之"的差异)
4. **必须遵守** §13.Y.4 三步骤 commit 流程
5. **必须登记** §11 待办:本 chain 特有的真实性校验缺失项

**违反本节 = §10 commit message 真实性核查红线 + §8 数据准确度原则 + §13.X 数据层同步纪律 三重复合违规**。
