# CLAUDE_CORE_RULES.md · 产业链投研核心准则(横跨所有赛道)

> **本文件是横跨所有产业链赛道的核心方法论提炼。适用于任何新增产业链时套用,例如 [产业链名称 1]/[产业链名称 2]/[产业链名称 3]/...等等**。
>
> 与 CLAUDE.md 区别:
> - **CLAUDE.md**(本仓仓库专属)包含具体数据字段、本仓历史案例、所有 commit 归档
> - **CLAUDE_CORE_RULES.md**(本文件)只含跨产业链通用规则 → 可作为模板复制到新仓库
>
> 用法:
> 1. 新建一个产业链仓时,复制本文件作为 CLAUDE.md 基础
> 2. 在 §17(本文件中预留的扩展章节)填具体赛道信息
> 3. 加上 §6.7.1 防御章节的具体 prompt 范例
> 4. 逐渐沉淀出本赛道专属的 §11 待办清单格式

## §1 数据真实性铁律(0 级原则,适用于所有数据改动)

### 总原则

数据用于投资研究,**准确性高于一切**。**绝不为了流程跑完而违反纪律**(详细见 §8 数据准确度优先原则)。

### 三条不可逾越红线

| 红线 | 内容 | 违反后果 |
|---|---|---|
| **CC 不许造数** | AI 永远不得自行编造财报、市占、缺口、产能、价格 | 数据真实性失守 |
| **来源不得虚标** | 任何数字引用必须可核验(`L1/L2/L3/L4` 单源采信,`L5` 需两家以上) | §6.2 红线 + §8.1 红线 |
| **未核实的必须显式标注** | 联网搜不到 → 标 `estimate` 或保留旧值,绝不能填看似精确的数字 | 掩盖不确定性 |

## §2 信源分级体系(L1-L5 + estimate)

适用所有"会变的事实":市占率、缺口率、CAGR、产能、财报、PE 分位 等。

| 等级 | 来源类型 | 采信规则 |
|---|---|---|
| **L1**(primary) | 财报 / 公告 / 交易所 / 巨潮(cninfo) | 单源直接采信 |
| **L2**(监管 / 协会) | 政府部委 / 行业协会 / 监管数据 | 单源直接采信 |
| **L3**(专业机构) | Prismark / IHS / 弗若斯特沙利文 / SEMI / Yole / Gartner / TrendForce 等 | 单源采信,标注机构名+报告名+日期 |
| **L4**(头部券商) | 华泰 / 东吴 / 国金 / 东北 / 东方证券 等 | 单源采信,**只提供"券商名+报告标题+发布日期+关键数字",禁止提供任何 PDF URL** |
| **L5**(财经媒体) | 新浪 / 东方财富 / 雪球 / 财联社 / 今日头条 | **需两家以上媒体同一数字才采信** |
| **L6**(非公开) | 雪球用户帖 / 匿名调研纪要 / 未具名信源 | **直接拒绝,不得引用** |

### 渲染层兼容(已存档四档)

页面渲染字典 `SOURCE_TIERS` 只支持四档:**primary / broker / media / estimate**。
- `L1` → primary
- `L2`/`L3`/`L4` → broker
- `L5` → media
- AI 主观推断(如六维分、估值初判) → estimate
- 新增数据点时按上表选择 tier;缺省时返回空字符串,向后兼容

## §3 数据写入流程(7 步)

1. **明确 6 元组**:stock code + name + segment + rank + barrier tier + trend + trendNote
2. **禁止引入新 stock**:只有 prompt 列表里的 stock,禁止造其他 stock
3. **禁止伪造 URL**:包含 cninfo / dfcfw / 慧博 / iFinD / 任何 HTTPS 链接——LLM 无能力核验 URL,会编造
4. **7 段式输出**:
   - 【1. 认证进展】 / 【2. 客户切换】 / 【3. 新进入者】
   - 【4. 技术产能壁垒】 / 【5. 品类归属】 / 【6. 未查到】 / 【7. 信源清单】
5. **L1-L6 信源分层**(§2)+ 标注 `L<数字>:<来源类型>+<公告/报告标题>+<日期>+<关键数字>`
6. **数字带口径**:每个百分比必须显式标注(占比 / 同比 / 市占率 / 产能利用率 / 毛利率 / 良率 / PE 分位),不允许裸 `%`
7. **trend 独立判定**:trend 判断主依据必须是 A 类信号(认证 / 客户验证 / 市占 / 产能 / 技术),B 类(营收/订单/利润)只辅助印证

## §4 占比字段核实三件套

任何"某 stock 某业务/产品占公司总营收 X%"类数字必须先走 **akshare `stock_zygc_em`** 实测核验再写入。

| 步骤 | 内容 |
|---|---|
| 1 | 先用 akshare `stock_zygc_em` `MAINOP_TYPE=2`(按产品)+ `MAINOP_TYPE=1`(按行业)拉取主营构成 |
| 2 | 与豆包返回的占比数字对比,偏差 >5pp 视为幻觉,丢弃改用 akshare |
| 3 | 偏差 ≤5pp 标注 "⚠️ 口径差异" 保留豆包数字 |

akshare 数据通路 = emweb.securities.eastmoney.com,Windows 代理可通;**返回 GBK 编码,必须设 `r.encoding='gbk'`**。

akshare 不替代"产能 / 产能利用率 / 产量"——这部分仍须读年报 PDF + 强制窄范围复核。

## §5 强措辞使用规范

涉及**强承诺性表述**时必须慎重,因为投资场景下"锁单""独家""框架协议""定点"等措辞一旦失实,会误导实盘下单决策。

| 强措辞 | 强制要求 | 整改模板 |
|---|---|---|
| **锁单 / 长期供货协议** | 必须有 L1 年报/季报/专项公告披露 | 不写 → 改"客户验证推进 / 行业景气方向" |
| **独家供应** | 必须有 L1 + L3 双源,**且**全球厂家分布数据支持 | 不写 → 改"卡口环节之一" |
| **框架协议 / 定点协议** | 必须有 L1 法定公告披露 | 不写 → 改"客户批量供货 / 客户认证可视" |
| **唯一 / 唯二 / 寡头** | 全球厂家分布证据支持 + 认证周期 ≥18 月 | 不写 → 改"头部厂商之一" |
| **量产**(对应"已通过认证 + 批量供货") | 需 L1 公告披露批量金额或销量,非单一行业传闻 | 弱 → 仅"试生产"或"送样验证" |

**典型教训**:某次 bid 验证发现某 stock 的 position 字段被误加"国内市占 70%"等具体数字,经 akshare 实测 + cninfo 公告核对后该数字无据,整改时**改写为定性描述**(具体数字保留在 position/investableReason 字段以便未来追溯),凡未核实精度数字归【6. 未查到】或"具体数字待人工核实"。

## §6 日期核实清单(针对"具体日期"的强约束)

LLM 在日期类任务中**极易编造看似合理的具体日期**(例如把"2026Q3 小批量出货"细化为"2026-06-01")。

凡涉及**以下场景**必须强制走 4 类禁止:

| 场景 | 典型幻觉 |
|---|---|
| L1 公告认证日期 | 笼统"2026Q3 小批量" → LLM 细化为"2026-06-01 获官方认证"(无 L1 原文依据) |
| 设备销售单价 | "单台设备售价 600 万元(溢价一倍)"(实际数据字段完全无此表述) |
| 占比具体小数 | "AI 占主营业务 43.20%" vs "主营占比 93.74%" 嵌套混乱 |
| 早年历史数据 | "2021/2022 营收 XXX"(实际 <某年才上市> 无该年完整数据) |

**4 类禁止规则**:

1. **禁止**:具体认证日期(2026-MM-DD 形式)→ 归【6. 未查到】
2. **禁止**:具体设备销售单价 / 补贴金额 / 锁单金额 → 归【6. 未查到】或"具体数字待人工核实"
3. **禁止**:占比小数扩展(如"全球市占 26.9%" 是否包括 AI 服务器细分) → 限定嵌套口径
4. **禁止**:上市前的早年历史数据 → 限定"3 年完整可追溯"

**自查清单**(豆包 prompt 中显式要求):

| 检查栏 | 要求 |
|---|---|
| 高风险栏 | 主动声明"无删除原值,全程遵守 4 类禁止规则" |
| 中风险栏 | 主动软化"待 L1 原文核实" |
| 低风险栏 | 直接引用 + 标"position 字段·estimate" |
| 可重算栏 | 标"baostock L1 实测" + 给出具体算法 |
| 【6. 未查到】栏 | 完整覆盖 4 类禁止全部风险点 |
| trend 字段诚实判定 | 不沿用原 trend 值,独立判定 |

## §7 dims6 评分五档表(景气六维通用模板)

各赛道评分时引用本表;**特殊赛道(亏损 / 高景气 / 国产替代临界)允许有约束的弹性,必须显式标注**。

### 标准 5 档表

| 维度 | 5 分 | 4 分 | 3 分 | 2 分 | 1 分 |
|---|---|---|---|---|---|
| **景气持续性** | 3 年以上需求驱动 + L3 覆盖 + 下游锁单 | 1-2 年明确需求 + L3/L4 + 部分锁单 | 需求存在但周期性强,无锁单 | 需求疲软,产能过剩 | 需求收缩,库存高企 |
| **业绩可见度** | 年报/季报可见明确订单/框架协议 + L1 | L4 订单预测 + 客户公开验证 | L4 预测但无客户确认 | 仅有 L5 媒体 | 无可见订单 |
| **政策确定性** | 国家重点支持目录 + 专项补贴 + L2 | 行业政策 + L2 | 政策中性 | 政策收紧 | 政策明确不利 |
| **供需紧张度** | 全球缺口 >30% + L3 | 缺口 10-30% + L3/L4 | 供需基本平衡 | 供给略过剩 | 严重过剩 |
| **估值性价比** | PE 分位 <30% + 成长赛道历史低位 | PE 分位 30-50% | PE 分位 50-70% | PE 分位 70-85% | PE 分位 >85% 或历史极高位 |
| **壁垒安全垫** | 物理卡口(全球 ≤3 家)+ 认证 ≥18 月 | 认证壁垒 6-18 月 + 国内唯一/领先 | 壁垒存在但竞品 ≥5 家 | 壁垒低,竞争激烈 | 无壁垒,完全竞争 |

### 打分原则

1. **每个维度打分必须有数据依据**,不得凭感觉
2. **barrier ≤1 时综合分封顶 60**(质地硬约束不可绕过)
3. **barrier=极高但 q1=false 时**,在 fourQuestions 里加 note 字段说明原因(壁垒≠卡口)
4. **每次打分后 dims6.asOf 记录打分日期**(`YYYY-MM`)
5. **亏损公司专项规则**(§7.1)+ **估算 PE 失真 → 用 PB + PS + 业绩拐点**

### §7.1 亏损/经营异常公司专项规则(触发时强制生效)

触发条件:`L1 利润表最近 4 季度中 ≥1 季度净利润为负`或`最近 2 年度均为净亏损`或`PE-TTM 长期为负 >30 天`或`主营营收 CAGR <-10%`。

强制规则:

- **valuation 维度独立成条 query,禁止用 PE-TTM,必须用 PB(MRQ) 历史分位 + 亏损收窄趋势 + PS 横向对比**
- **query 前 baostock/akshare 预喂 L1 实证数据**(净利润正负 + 营收同比方向 + 实际 CAGR)
- **格式合规 ≠ 内容真实**:任何高风险维度的数字必须做 baostock/akshare 抽查验证
- **高风险数字特征识别**:精确小数点 / 整数百分比 / 全球整齐分布 / 单一信源覆盖 10+ 公司 → 必须要求豆包标注具体信源章节或列入【未查到】

## §8 数据准确度优先原则(0 级原则)

### 核心三原则(按优先级)

1. **数据准确度 > 流程完成** — 宁可任务不完成、数据保持上一稳定状态,也**绝不能为了"让流程跑完"而手动改字段或注入违规数据**
2. **严格遵守项目纪律** — 所有 CC 操作必须符合本文件 + §7 流程
3. **绝不为了流程跑完而违反纪律** — CC 推荐的所有方案必须先通过 §1 红线审查

### 反模式(禁止)

- ❌ "为了让 R-N 数据注入项目,CC 可以小改字段" — 违反 §1 + §8.1
- ❌ "手动修违规 URL 比重新跑模型快" — 违反 §1 + §8.3
- ❌ "流程上需要 commit,所以降低合规要求" — 违反 §8.1
- ❌ "用户可能不耐烦,所以走捷径" — 违反 §8.3
- ❌ "CC 代用户判断'哪些已修正'以节省 token" — 违反 §3.7 + §8.2
- ❌ "既然上一轮截断了,就下一轮只针对剩余违规" — 违反 §3.7(必须完整扫描)

### 验证清单(每次推荐方案前必走)

```
□ 1. 这个方案是否会让 CC 修改 LLM 返回的字段?
   - 是 → ❌ 拒绝(§1.2)
   - 否 → 继续 □ 2
□ 2. 这个方案是否会注入违反 §3.3 的字段(如 URL)?
   - 是 → ❌ 拒绝(§3.3)
   - 否 → 继续 □ 3
□ 3. 这个方案是否会让数据准确度降低?
   - 是 → ❌ 拒绝(§8.1)
   - 否 → 继续 □ 4
□ 4. 这个方案是否"为了流程跑完"而设计?
   - 是 → ❌ 拒绝(§8.3)
   - 否 → ✅ 可以推荐
```

## §9 barrier 维度跨环节引用禁令(<跨环节引用教训抽象代号>)

**barrier 维度禁止跨环节引用同业,必须严格在 stock **本细分赛道内**引用同业竞品**。

### 错误示范

某 stock 主营 `<主环节 idx>(<本细分赛道>)`,barrier 引用理由中**误引用**:
- 上游 `<上游环节 idx>(<上游细分>)` 厂商作为同业竞品
- 下游 `<下游环节>(<下游细分>)` 厂商作为同业竞品
- 隔环节 `<平行环节 idx>(<平行细分>)` 厂商作为同业竞品

### 正确做法

- **stock 主营 idx 内** 引用同业竞品(同环节 → 同环节内细分赛道 → 同细分赛道内同业厂商)
- 不要因为"上下游关联"或"同属一大行业"而模糊跨界
- 案例:某次 bid 验证发现某 stock 在 barrier reason 中误将"上游 X 厂商"列为同业竞品,整改后**严格限定在主营 idx 细分赛道内**,删除跨环节厂商名

### 强制 action

- 任何 reason 字段中提及"同业竞品 / 同业竞争者"必须**限定在该 stock 所在主营 idx 细分赛道内**
- 不可抗辩为"上下游同属 <产业链名称> 大行业,所以可互相引用"(这是红线)
- ack/diff 检查:`grep -E "同业竞品|同 segment" reason 文本` 必须命中主营 idx 内的厂商名,不允许命中跨环节 idx 内厂商名

## §10 commit message 真实性核查要求(<数据治理场景失真教训抽象代号>)

### 触发场景

任何**写入数据文件**的 commit,撰写 commit message 时若使用以下表述:

- "已完成" / "已补全" / "已有 reason 基础上维持" / "修复完成" / "全部通过"
- "N 处变更" / "M 项修改" / "X 个 stock 已校准"

### 强制要求

**commit 前必须先 grep 实际文件状态确认**,**不能凭印象撰写**:

1. **步骤 1**:用 awk/grep/python 提取实际文件状态(例如 stock 列表、dims6 reason 完整度、tier 字段值)
2. **步骤 2**:与 commit message 中声称的"已完成 / 已补全"对照
3. **步骤 3**:若发现不一致,**要么修正 commit message,要么重新跑 batch**
4. **步骤 4**:若已经失真而 commit 已 push,在下个 commit 中显式说明"上一 commit 标题描述与实际不符"并修正

### 禁止语句(直接 fail)

- ❌ "已有 reason 基础上维持"(实际文件无 reason)
- ❌ "8/8 reason 补全"(实际只有 7/8)
- ❌ "治理完成"(实际还有 3 个维度未审)
- ❌ "验证通过"(实际 baostock 拉取失败)

### 失真案例归档

| 失真类型 | 案例 | 后果 |
|---|---|---|
| 标题叙述与文件状态不符 | commit 标题写"已有 reason",实际 commit 时无 reason | 误导后续维护者,不知道该补哪里 |
| 完成数夸大 | commit 标题写"8/8 完成",实际只有 7/8 | 后续治理时跳漏 1 处 |
| 修复范围虚标 | commit 标题写"N 处",实际 git diff 只有 M 处(M < N) | 信源追溯失效 |

### 防御机制升级

- **commit 前置 gate**:用 `python scripts/page_audit.py` 第【N】项自动核对提交声称
- **dims6Audit 字段**:任何 `score=N` 的字段必须有 `reason` + `reviewedAt` + `reviewer`,缺一不可
- **每周回放 grep**:每周对前 10 个数据 commit 做"声称 vs 实际" diff,如有失真立即 fail 并修正

## §11 数据自查纪律(任何数据变更 commit 必须输出)

### 触发条件

- 任何涉及数据字段的 commit(新增 / 修改 / 删除)
- 脚本生成或覆盖数据文件后
- 发现数据异常或 bug 修复后
- 阶段交界验收时

### 自查报告格式

```
=== 数据自查报告 · <commit 编号> ===

【已知正确】
- 列出本次确认无误的数据项 + 验证依据(来源/方法)

【已知错误/异常】
- 列出发现的问题 + 根因 + 是否已修复 + 修复方法

【待核实(人工抽查)】
- 列出无法自动验证、需要人工核对的项目
- 必须指明:哪几只 stock + 去哪里核对(同花顺/券商研报/公司公告)+ 核对路径

【数据完整性统计】
- 总数 / 有效数 / null 数 / 异常数
- 各关键字段覆盖率(pe_ttm / pePercentile / source / asOf)
```

### 硬约束

- 自查报告发现的【已知错误】**必须在下一个 commit 修复**
- 【待核实】项目**必须在阶段交界验收前完成人工核对**
- 自查报告**永久记录**在 commit message 或 commit 后回复里
